# Plan

AVA v7.4.1 can introduce an explicitly opt-in community deployment directory that gives operators and service providers useful public visibility without turning installations into telemetry sources. The safest first release is a moderated, card-based directory backed by a separate registry service; a world map and richer discovery can follow after the consent, removal, and abuse controls have been proven.

## Scope

- In: an operator-initiated CLI registration flow, explicit publication consent, email verification, moderation, public organization cards, update/removal commands, privacy and retention rules, and a separately hosted registry API and directory.
- Out: automatic installation discovery, IP geolocation, precise server locations, background telemetry, donation-gated placement, popularity ranking, an interactive world map in the first release, and registry implementation in the v7.4.0 release.

## Findings and decisions

- The idea offers a healthy value exchange: participating operators receive free visibility while AVA gains credible, consented evidence of adoption.
- `demo.agent6789.com` currently serves the Admin UI demonstration. The public directory and its write API should use separate trust boundaries, such as `community.agent6789.com` and `registry.agent6789.com`.
- Registration must be a deliberate command. AVA must not infer location from an IP address or publish an installation merely because it is reachable.
- The first release should use searchable cards and country/service-area filters. A large world map is attractive, but adds geocoding, accessibility, clustering, privacy, and maintenance work that is not necessary to validate the concept.
- "Verified" should mean that the registrant controls the submitted email or domain. It must not imply that AVA endorses the deployment, business, or service quality.
- Donations and directory participation must remain separate. Sponsors may receive a clearly labeled sponsor badge, but payment must not be required for listing or used to rank ordinary entries.

## Proposed operator experience

1. The operator runs `agent community join` (`agent register` may be retained as a discoverable alias).
2. The CLI explains exactly what will become public and links to the privacy policy.
3. It collects a display name or business name, contact email, deployment category, country, service areas, website, short description, and optional logo.
4. The CLI renders the proposed public card and asks for explicit publication consent. Newsletter or project-update consent is a separate, unchecked choice.
5. The registry emails a verification link. Verified submissions enter moderation instead of publishing immediately.
6. The operator can run `agent community status`, `agent community update`, or `agent community remove` at any time. Removal must be as easy as registration.

## Data and consent contract

Public fields:

- Display or business name
- Deployment category: end user, integrator, service provider, or community project
- Country and self-declared service areas
- Website and optional public contact address
- Short description and optional logo
- Verification state, registration date, and optional sponsor badge

Private registry fields:

- Verification email
- Consent version and timestamp
- Moderation history
- Hashed management token and abuse-prevention metadata

Never collect or publish:

- API keys, PBX credentials, call metadata, caller data, prompts, transcripts, agent configuration, public IP address, exact server coordinates, or usage telemetry

Consent must be specific, informed, and revocable. The preview shown before submission must match the public card. Records should be reconfirmed every 12 months; expired records should be hidden until reconfirmed. Removed records should leave the public directory immediately and be purged from normal registry storage within 30 days, except for the minimum abuse/audit record required by policy.

## Service and security design

- Host the write API separately from the demo Admin UI and publish a read-only directory feed to the public site.
- Use states `pending_email`, `pending_review`, `published`, `suspended`, `removed`, and `expired`.
- Return a one-time management token after registration, store only its hash server-side, and save the client token to `.agent/community-registration.json` with mode `0600`. Provide verified-email recovery.
- Suggested API surface: `POST /v1/registrations`, `GET/PATCH/DELETE /v1/registrations/{id}`, `POST /v1/registrations/{id}/verify`, and a separate moderated public listing endpoint.
- Rate-limit submission, verification, recovery, and logo upload endpoints. Add CAPTCHA or proof-of-work only if abuse justifies the added friction.
- Accept only bounded PNG, JPEG, or WebP logos; verify file signatures, decode and re-encode images, strip metadata, generate fixed-size derivatives, randomize stored names, and never accept SVG or serve user uploads from the application origin.
- Validate website schemes and render external links with `noopener noreferrer`. Apply content moderation, duplicate detection, and an abuse-report path before general availability.
- Document that public entries are self-reported. Domain/email verification proves control only.

## Milestones

- [ ] Milestone 1 — policy and manual pilot: publish privacy/acceptable-use terms, finalize the public fields and consent text, create a manual submission form, and moderate 5–10 willing community entries.
- [ ] Milestone 2 — v7.4.1 registry foundation: build the isolated registry API, verification email, moderation queue, card directory, CLI join/status/update/remove flow, token recovery, removal workflow, and operator documentation.
- [ ] Milestone 3 — post-v7.4.1 discovery: evaluate pilot data, then add map visualization, richer filters, automated logo processing, localization, and sponsor badges only if the directory has enough entries to justify them.

## Test and verification plan

- [ ] Unit-test CLI validation, preview/consent gating, token persistence permissions, API schema validation, status transitions, expiry, and removal.
- [ ] Integration-test email verification, moderation approval/rejection, update recovery, immediate unpublishing, and scheduled purge/reconfirmation jobs.
- [ ] Security-test rate limits, enumeration resistance, management-token hashing, malicious URLs, oversized/polyglot image files, metadata stripping, stored-XSS payloads, and cross-origin upload isolation.
- [ ] Accessibility-test directory cards, filters, keyboard navigation, focus handling, error messages, and the no-map fallback before adding a map.
- [ ] Verify that a fresh or upgraded AVA installation makes no registry request until an operator explicitly runs the join command.
- [ ] Run a deletion drill that proves a published card disappears immediately and its retained data follows the documented purge policy.
- [ ] Pilot with both individual operators and businesses in multiple countries, then review confusion around "verified," privacy expectations, moderation workload, and listing usefulness.

## Complexity and regression risk

| Area | Complexity | Regression risk | Notes |
| --- | --- | --- | --- |
| CLI registration and local token | Medium | Low | Isolated commands; no call-path dependency |
| Registry API, verification, and moderation | Medium–High | Low to AVA, Medium operational | New service and abuse surface |
| Public card directory | Medium | Low | Read-only consumer of moderated records |
| Logo upload | Medium–High | Medium security | Requires strict transformation and isolated hosting |
| Interactive world map | High | Medium | Defer until listing volume and privacy model are proven |

## Open questions

- Should v7.4.1 accept personal/individual listings, or only organizations and public community projects?
- Is verified email plus moderation sufficient, or should business listings require domain verification?
- Is the recommended 12-month reconfirmation and 30-day post-removal purge acceptable for the published privacy policy?
